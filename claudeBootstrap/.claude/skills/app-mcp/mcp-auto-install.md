# MCP Auto-Install Pattern: Signed API Key with Embedded Workspace ID

## Problem

The standard MCP multi-tenant pattern requires **two headers** on every `tools/call`:
- `workspace-id` (or `workspace-slug`): identifies the calling workspace
- `mcp-api-key`: authenticates the request

This is cumbersome for MCP clients (agents, external tools). Ideally, a single credential should suffice.

## Solution: HMAC-Signed Self-Contained API Key

The API key itself encodes the workspace ID and the webhook URL needed to fetch the tenant's config. The key is HMAC-signed by the central workspace's secret, making it unforgeable.

### Key Format

```
base64url(JSON payload) + "." + hmac_sha256(JSON payload, appSecret)
```

**JSON payload:**
```json
{
  "wid": "W-pqCPO",
  "url": "https://api.studio.prisme.ai/v2/workspaces/W-pqCPO/webhooks/MyApp.getConfig"
}
```

- `wid`: the workspace ID of the tenant (installing workspace)
- `url`: the full URL to the `getConfig` webhook of the app instance in that workspace

The workspace ID is in cleartext (base64) — it's not a secret. The HMAC signature proves the key was generated by the central workspace (unforgeable without `appSecret`).

## Architecture

```
                          ONE-TIME (install)
 ┌──────────────────┐     ┌──────────────────────────────┐
 │  Tenant          │     │  Central MCP Workspace        │
 │  Workspace       │     │                               │
 │                  │  1  │  onInstall                    │
 │  installs app ───┼────>│    │                          │
 │                  │     │    ▼                          │
 │                  │  2  │  POST /webhooks/generateKey   │
 │  onInstall ──────┼────>│    │ signs {wid, url}         │
 │                  │     │    │ with appSecret            │
 │                  │  3  │    ▼                          │
 │  stores key  <───┼─────│  returns signed API key       │
 │  in config       │     │                               │
 └──────────────────┘     └──────────────────────────────┘

                       EVERY CALL (runtime)
 ┌──────────────────┐     ┌──────────────────────────────┐
 │  MCP Client      │     │  Central MCP Workspace        │
 │  (agent, curl)   │     │                               │
 │                  │  1  │  POST /webhooks/mcp           │
 │  tools/call ─────┼────>│  header: mcp-api-key          │
 │  (1 header only) │     │    │                          │
 │                  │     │    ▼ verifyAndDecodeKey()      │
 │                  │     │    │ extracts wid + url        │
 │                  │     │    ▼                          │
 │                  │  2  │  GET {url} (getConfig)        │
 │                  │     │    │ fetches tenant config     │
 │                  │     │    ▼                          │
 │                  │  3  │  routeToolCall                │
 │  result <────────┼─────│    returns tool result        │
 └──────────────────┘     └──────────────────────────────┘
```

## Key Automations

### 1. `generateKey.yml` — Central webhook for key generation

**Location:** Central workspace only.
**Trigger:** `when.endpoint: true`
**Role:** Receives `{workspaceId, getConfigUrl}` via POST, signs with `config.appSecret`, returns the signed key.

This is the **only place** where `appSecret` is used for signing. App instances never have access to the secret.

**Auto-generated `appSecret` (self-healing).** If `appSecret` is not a real resolved value (fresh connector, nobody pasted a value into the Secrets section), `generateKey` generates a 256-bit random secret via the `generateRandomSecret` Custom Code function, persists it to the central workspace's `/security/secrets` (the `PATCH` is NOT terminal — unlike `set: config type: merge` — so the run still returns its response), and signs the current call with the in-memory value. Subsequent calls read the now-persisted `{{config.appSecret}}`. A human-set value is always respected (the check skips generation). See "Known limitation: concurrent first-install race" below.

**Why the "is it set?" check goes through Custom Code (not `!{{config.appSecret}}`).** On the central workspace an UNSET secret does NOT make `{{config.appSecret}}` empty — it resolves to the literal binding string `{{secret.appSecret}}`, which is **truthy**. A plain `!{{config.appSecret}}` guard is therefore always false, so the self-heal never runs and `generateKey` silently signs every key with the literal string as the HMAC key (a stable but bogus secret — keys verify against each other but the appSecret is never really provisioned). And a `matches "{{"` condition can't be used either: it crashes with `InvalidVariableNameError` on a literal-binding value. The robust detector is the `isUsableSecret` Custom Code function (`usable = non-empty && no "{{"`).

```yaml
slug: generateKey
when:
  endpoint: true
do:
  - set:
      name: effectiveSecret
      value: '{{config.appSecret}}'
  - Custom Code.run:
      function: isUsableSecret
      parameters:
        value: '{{config.appSecret}}'
      output: secretCheck
  - conditions:
      '!{{secretCheck.usable}}':
        - Custom Code.run:
            function: generateRandomSecret
            output: generatedSecret
        # ... guard generatedSecret.error -> 500 ...
        - auth:
            workspace: true
            output: wsJwt
        - fetch:
            url: '{{global.apiUrl}}/workspaces/{{global.workspaceId}}/security/secrets'
            method: PATCH
            headers:
              Authorization: Bearer {{wsJwt.jwt}}
            body:
              appSecret:
                value: '{{generatedSecret}}'
                description: HMAC secret, auto-generated on first key request.
            output: persistResp
            outputMode: detailed_response
            emitErrors: false
        # ... guard persistResp.status non-2xx -> 500 ...
        - set:
            name: effectiveSecret
            value: '{{generatedSecret}}'
  - Custom Code.run:
      function: generateSignedKey
      parameters:
        workspaceId: '{{body.workspaceId}}'
        getConfigUrl: '{{body.getConfigUrl}}'
        secret: '{{effectiveSecret}}'
      output: signedKey
  - set:
      name: response
      value:
        mcpApiKey: '{{signedKey}}'
output: '{{response}}'
```

### 2. `onInstall.yml` — App instance auto-setup

**Trigger:** `workspaces.apps.installed`, `workspaces.apps.configured`
**Role:** When the app is installed in a tenant workspace, calls `generateKey` on the central workspace and stores the result.

**Key variables available in onInstall context:**
- `{{global.workspaceId}}` — ID of the tenant workspace (the one installing the app)
- `{{run.appInstanceSlug}}` — slug of the app instance (falsy if self-trigger on central workspace)
- `{{global.apiUrl}}` — platform API base URL (already includes `/v2`)

**Guards:**
- `!{{run.appInstanceSlug}}` → skip if self-trigger (not an app instance)
- `{{config.mcpApiKey}} matches "\\."` → skip if key already generated

**Flow:**
1. Build `getConfigUrl` = `{{global.apiUrl}}/workspaces/{{global.workspaceId}}/webhooks/{{run.appInstanceSlug}}.getConfig`
2. POST to `{{global.apiUrl}}/workspaces/slug:<app-slug>/webhooks/generateKey` with `{workspaceId, getConfigUrl}`
3. Store returned key + MCP endpoint URL in `config` (merge)

**Config fields set by onInstall:**
- `mcpApiKey`: the signed key (for the MCP client to use)
- `mcpEndpoint`: the MCP endpoint URL (convenience for the user)

### 3. `mcp.yml` — MCP endpoint with signed key auth

**Trigger:** `when.endpoint: true`
**Role:** Central MCP endpoint. For `tools/call`, decodes the signed key to extract the workspace ID and getConfig URL, then fetches the tenant config and routes to the tool.

**Important architectural decisions:**

- **`tools/call` bypasses MCP Core** — we handle auth, config lookup, and tool routing directly. MCP Core's internal `break` on API key validation conflicts with DSUL error handling. MCP Core is only used for `tools/list` and `initialize`.
- **Nested `conditions` with `default`** instead of `break` — avoids the `break: { scope: all }` propagation issue in DSUL `try/catch` blocks.
- **Session cache** — tenant config is cached in `session.tenantConfig[workspaceId]` with a 10-minute TTL to avoid repeated getConfig calls.

### 4. `getConfig.yml` — Tenant config webhook

**Trigger:** `when.endpoint: true`
**Role:** Runs in the tenant workspace's app instance context. Returns the app instance config to the central MCP endpoint.

### 5. Custom Code functions

**`generateSignedKey(workspaceId, getConfigUrl, secret)`:**
```js
const crypto = require('crypto');
const data = JSON.stringify({ wid: workspaceId, url: getConfigUrl });
const payload = Buffer.from(data).toString('base64url');
const signature = crypto.createHmac('sha256', secret)
  .update(data).digest('hex');
return payload + '.' + signature;
```

**`verifyAndDecodeKey(apiKey, secret)`:**
```js
const crypto = require('crypto');
const parts = (apiKey || '').split('.');
if (parts.length !== 2) return { error: 'Invalid key format' };
const data = Buffer.from(parts[0], 'base64url').toString();
const expected = crypto.createHmac('sha256', secret)
  .update(data).digest('hex');
if (parts[1] !== expected) return { error: 'Invalid signature' };
const parsed = JSON.parse(data);
return { workspaceId: parsed.wid, getConfigUrl: parsed.url };
```

**`generateRandomSecret()`** — parameterless; used by `generateKey` to self-heal an empty `appSecret`:
```js
const crypto = require('crypto');
return crypto.randomBytes(32).toString('hex');
```

**`isUsableSecret(value)`** — used by `generateKey` to decide whether `appSecret` is a real resolved secret vs an unresolved binding literal (`{{secret.appSecret}}`):
```js
const v = (value === null || value === undefined) ? '' : String(value);
return { usable: v.length > 0 && v.indexOf('{{') === -1 };
```

## index.yml Config Schema

### Config schema (visible to app instances)

```yaml
config:
  schema:
    mcpEndpoint:
      type: string
      title: MCP Endpoint
      readOnly: true        # auto-generated, not user-editable
    mcpApiKey:
      type: string
      title: MCP API Key (auto-generated)
      readOnly: true        # auto-generated, not user-editable
```

### Config value (central workspace defaults)

```yaml
  value:
    appSecret: '{{secret.appSecret}}'   # only resolved on central workspace
    mcpTools:                           # tool definitions for MCP Core
      - name: myTool
        ...
```

**Important:** Do NOT put `mcpApiKey` in `config.value` — otherwise `{{config.mcpApiKey}}` is truthy on fresh instances (the template string itself is truthy even when unresolved), and `onInstall` skips key generation.

### Secrets (central workspace only)

```yaml
secrets:
  schema:
    appSecret:
      type: string
      title: App HMAC Secret
      ui:widget: password
```

### Labels

Include `importFrom:<workspaceId>` to make the workspace installable as an app.

## Gotchas & Lessons Learned

### 1. `config.appSecret` is workspace-scoped

`config.value.appSecret: '{{secret.appSecret}}'` resolves to the **central workspace's** secret only on the central workspace. On app instances, `{{secret.appSecret}}` resolves to the tenant workspace's secret (which is empty).

**Consequence:** Key generation MUST happen on the central workspace via the `generateKey` webhook. The `onInstall` automation on the instance calls the central workspace — it cannot sign locally.

**This is also why auto-generation of `appSecret` lives in `generateKey`, not in an install event:** there is no clean "central workspace created" lifecycle event, and on a tenant `{{config.appSecret}}` is *also* empty (it resolves the tenant's own empty secret) — so an event-triggered provisioner could not tell central from tenant apart. `generateKey` only ever runs on the central workspace, so it is the natural (and only safe) place to self-heal the secret.

### 2. Don't put auto-generated config in `config.value`

If `config.value.mcpApiKey: '{{secret.mcpApiKey}}'` is set, new instances inherit this template string. Even when the secret is empty, `{{config.mcpApiKey}}` evaluates as truthy (it's a non-empty string), causing `onInstall` to skip key generation.

**Fix:** Omit `mcpApiKey` from `config.value`. Only set it via `onInstall` (config merge).

### 3. `break: { scope: all }` in `try/catch` causes `{"break":{}}` errors

Using `break: { scope: all }` inside deeply nested conditions within a `try` block causes the break to propagate as an exception caught by the outer `catch`. The catch receives `$error = {"break":{}}` with an empty message.

**Fix:** Use nested `conditions` with `default` branches instead of `break` for flow control. Reserve `break` for simple early exits in non-try contexts.

### 4. MCP Core's API key validation uses internal `break`

When MCP Core validates the `mcp-api-key` header against its configured `apiKey`, a mismatch causes an internal `break` that propagates up as an unhandled exception in the calling automation.

**Fix:** For `tools/call`, bypass MCP Core entirely — handle JSON-RPC response construction manually. Use MCP Core only for `tools/list` and `initialize` (which don't require API key validation).

### 5. `global.apiUrl` already includes `/v2`

`{{global.apiUrl}}` resolves to e.g. `https://api.studio.prisme.ai/v2`. Do not append `/v2` again when building webhook URLs.

### 6. Webhook URL uses app instance slug, not app name

The webhook URL for an app instance automation is:
```
{{global.apiUrl}}/workspaces/<workspaceId>/webhooks/<appInstanceSlug>.<automationSlug>
```

The `appInstanceSlug` is available as `{{run.appInstanceSlug}}` in `onInstall`. It is NOT the workspace name with spaces — it's the camelCase slug assigned by Prisme (e.g., `DebugMCPAPIKeyWwithWorkspaceID`).

### 7. Central workspace MCP endpoint URL uses automation slug

The MCP endpoint URL is simply:
```
{{global.apiUrl}}/workspaces/slug:<workspace-slug>/webhooks/mcp
```

Where `mcp` is the **automation slug**, not the app name.

### 8. Known limitation: concurrent first-install race

Auto-generation of `appSecret` is idempotent but not atomic. If two tenant installs hit `generateKey` near-simultaneously on a brand-new connector (before the first `PATCH /security/secrets` lands), both generate different secrets and the last `PATCH` wins. The tenant whose key was signed with the losing secret will fail `verifyAndDecodeKey` ("Invalid signature") at `tools/call` time.

This is **low probability** (the window is a few hundred ms, only on the very first provisioning) and **self-recovers**: reconfiguring that app instance re-fires `onInstall`, which re-issues a key against the now-persisted secret. To avoid the race entirely on a high-traffic launch, set `appSecret` manually before the first install. A re-GET after the `PATCH` does not close the window, so we don't add one.
